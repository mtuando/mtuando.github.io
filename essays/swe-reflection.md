Throughout my time in ICS 314 I have come to realize that software engineering is also about managing complexity, working together effectively and building systems that solve real problems. While this course taught me to build web applications using available tools and frameworks, the deeper lessons I learned go beyond web development. 

# Issue-Driven Project Management

<u>Issue-Driven Project Management (IDPM)</u> is an implementation of Agile principles that we used extensively in ICS 314. In IDPM all work is tracked through issues or tasks. These are typically managed on platforms like GitHub. Each issue represents a unit of work that is clearly defined with acceptance criteria. It is assigned to team members. Tracked through development and testing, the team meets regularly to review progress work as needed and plan the next iteration. It kept things transparent: everyone knew what was done, what was stuck, and what was coming up next. The real value is to be able to organize collective works. It creates accountability, visibility into progress and a clear definition of "done" for each piece of work. It’s a way to handle the chaos of a complex project.

# Configuration Management

<u>Configuration Management (CM)</u> is the practice of tracking and controlling changes to a software system over time. In the course, we learned the practice from using GitHub to version control our code. It involves specifying dependencies in package files like ```package.json```, to  writing test scripts that automate the build and deployment process. The benefit is that any team member can clone the repository and run one command to get a working development environment, and no "it works on my machine" excuses allowed. It helps maintain sustainability of our systems.

# Conclusion

At the end of the day, these qualities not only teach me proper skills that I may see on the job, but to also build habits and help coordinate a project that has a lot of areas to work on. Configuration Management handles the organization of our project’s code, while IDPM acts similar to a task management tool. Along with various not mentioned concepts, this course has shown me that being a "software engineer" isn't about mastering a specific language—it's about having a reliable system to handle the complexity of building anything great.
